## 슈핏케어 보안 하드닝 가이드

### 1) 저장 최소화 원칙
- 고객 대화 원문은 장기 보관하지 않음
- 운영 로그는 원문 대신 `message_hash` + `message_len`만 저장
- 배포 저장소에는 `data/`, `.db`, `.log`, `.env`를 절대 올리지 않음

### 2) 기본 운영 규칙
- `shop_id` / `policy_version`를 모든 이벤트에 남겨 추적성 확보
- 장애 로그는 개인정보 포함 금지
- 디버깅이 필요해도 원문 재수집은 승인 후 일시적으로만 수행

### 3) 접근 통제
- 운영 서버 관리자 계정 MFA 필수
- 저장소 권한 최소화(읽기/쓰기 분리)
- 외부 공유 시 데이터 샘플은 마스킹 후 제공

### 4) 키/비밀정보
- API 키는 환경변수로만 관리
- 코드/문서/채팅에 비밀키 복붙 금지
- 키 교체 주기 운영(최소 분기 1회)

### 5) 보관/파기
- 로그/세션/실험데이터는 보관 기한 설정 후 자동 삭제
- 발표/테스트용 데이터는 발표 종료 후 즉시 정리

### 6) 사고 대응(간단 SOP)
1. 외부 노출 의심 즉시 키 폐기 및 회전
2. 공개 저장소/링크 차단 및 접근권한 축소
3. 유출 범위 점검(기간/필드/대상)
4. 재발 방지 조치(룰/자동검사) 반영
